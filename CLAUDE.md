## What this repo is

Production **deployment stack** for the SIEM Source Onboarding Tracker. The app source lives in the separate `siem-tracker` repo; nothing is built here â€” the stack runs the published public image `ngmaibulat/usiem-tracker:latest` (versioned `vX.Y.Z` / timestamp tags exist in the registry for rollback). There is no application code, no package manager, no test suite.

**This repo is public.** `.env` and `nginx/certs/*.pem` are gitignored secrets; docs use placeholders (`<prod-host>`, `<prod-user>`). Never commit real hostnames, credentials, or certs.

## Commands

All run from the repo root (default `docker-compose.yml`, no `-f` needed):

```bash
docker compose pull                # fetch the published image
docker compose run --rm migrate    # one-shot Prisma migrations, idempotent (profile keeps it out of `up`)
docker compose up -d               # bring the stack up
```

The same three commands are both first deploy and update procedure. On schema changes, always `migrate` **before** `up -d app`. Verify: `docker compose ps` (app has a HEALTHCHECK), `curl -k https://localhost/api/health`.

Never run on prod: `docker compose down -v`, `docker volume rm <project>_postgres_data`, `docker system prune --volumes` â€” these destroy the database.

## Architecture

Topology: **nginx (TLS, only service publishing ports 80/443) â†’ app â†’ postgres / redis / squid / meilisearch**, all on the internal compose network.

The non-obvious part is that **nginx's config and certs are generated by the app**, not by files in this repo:

- The app's Initial Configuration Wizard (`/setup`, runs on a fresh install) writes `app.conf` into the `proxy_conf` volume and `cert.pem`/`key.pem` into `proxy_certs`; nginx mounts both read-only (`nocopy` prevents the image's default config from seeding the volume).
- On first boot the app entrypoint seeds a bootstrap HTTP-only config so the wizard is reachable on port 80 before any cert exists; nginx's `command` waits for that file to appear. After the wizard finishes: `docker compose exec nginx nginx -s reload`.
- Consequently `nginx/conf.d/app.conf` and `nginx/certs/` in this repo are **reference/legacy only** â€” not mounted by the compose file.

Other volume contracts worth knowing:

- `app_data` (SQLite connection registry with encrypted creds, FR-31) and `app_backups` must survive recreation; they are deliberately on separate volumes from `proxy_conf`/`proxy_certs` so nginx never mounts anything containing secrets.
- Log tailing: nginx writes real files to `nginx_logs`, squid to `squid_logs`; the app mounts both read-only and its `/admin/logs` viewer tails them. The squid entrypoint override exists solely to make those logs world-readable.
- `squid` is the app's egress proxy (FR-32) â€” all outbound HTTP/HTTPS goes through it via `HTTP_PROXY`/`HTTPS_PROXY`; `NO_PROXY` keeps intra-stack traffic direct. The app degrades gracefully if squid is down.
- `meilisearch` is provisioned but **not yet wired into the app** (no `MEILI_*` code) â€” it still requires `MEILI_MASTER_KEY` because it runs with `MEILI_ENV=production`.

Environment: `cp example.env .env`. Required: `APP_URL`, `TARGET_DATE`, `MEILI_MASTER_KEY`; `APP_SECRET` is optional (the wizard generates and stores one). `DATABASE_URL`/`REDIS_URL` are hardcoded in the compose file to internal service names â€” only `POSTGRES_PASSWORD` is parameterized. Optional `CONTROL_DATABASE_URL` moves the control plane (identity/auth/audit) to a separate Postgres; unset, it lives in the `control` schema of the same DB. Migrations apply to **both** planes.
